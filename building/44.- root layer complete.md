# ENGINEERING_PLAYBOOK.md

```markdown
# byDebut Engineering Playbook

**Version:** 1.0

**Status:** Official

**Audience:** Engineers, AI Coding Assistants, Technical Reviewers

**Repository:** byDebut Company Brain

**Owner:** CTO / Founder

**Last Updated:** July 2026

---

# Purpose

This document defines how engineering work is performed at byDebut.

Unlike the Company Constitution, which defines why the company exists, or the Product Operating System, which defines how the platform behaves, the Engineering Playbook defines how software is built.

Every engineer and every AI coding assistant must follow this document.

---

# Engineering Philosophy

We optimize for:

- correctness
- maintainability
- readability
- scalability
- security
- traceability

Never optimize for writing code quickly.

Optimize for writing code that still makes sense five years from now.

---

# Engineering Principles

Every implementation should be:

- modular
- reusable
- testable
- observable
- documented
- extensible

Avoid clever code.

Prefer obvious code.

---

# Company Brain Authority

Before implementing anything, read documentation in this order:

1. README.md

2. COMPANY_BRAIN_INDEX.md

3. AI_CONTEXT.md

4. PROJECT_MANIFEST.md

5. Company Constitution

6. Mission & Vision

7. Hackathon Compliance

8. Platform Architecture

9. Product OS

10. Product Principles

11. Standards

12. Current Sprint

13. Current Task

If documents conflict, the higher authority document always wins.

---

# Development Workflow

Every feature follows the same lifecycle.

Architecture

↓

Review

↓

Approval

↓

Implementation

↓

Testing

↓

Documentation

↓

Code Review

↓

Merge

↓

Deployment

Never skip steps.

---

# Branch Strategy

Main

Production-ready code only.

Develop

Integration branch.

Feature Branches

One feature per branch.

Examples:

feature/story-passport

feature/story-portal

feature/authentication

feature/reader-dashboard

Never implement multiple unrelated features in one branch.

---

# Sprint Strategy

Each Sprint should contain:

Objectives

Scope

Acceptance Criteria

Testing Plan

Known Limitations

Future Work

Everything outside the Sprint remains out of scope.

---

# Coding Standards

Language

TypeScript

Strict Mode enabled.

Avoid "any".

Prefer explicit types.

Prefer interfaces over loose objects.

---

# Naming

Use descriptive names.

Avoid abbreviations.

Examples:

StoryPassport

StoryLineage

CreativeWorkspace

ReaderDashboard

Avoid:

sp

obj

tmp

data2

foo

---

# Folder Organization

Organize by domain.

Preferred structure:

app/

modules/

components/

services/

hooks/

types/

lib/

utils/

Never organize only by pages.

---

# React Principles

Prefer Server Components.

Use Client Components only when required.

Keep components small.

One responsibility per component.

Avoid deeply nested components.

---

# State Management

Keep state as local as possible.

Use React Server Components when appropriate.

Avoid unnecessary global state.

Use Context only when justified.

---

# API Principles

Every endpoint should:

have one responsibility

validate inputs

return typed responses

handle errors gracefully

avoid leaking internal implementation

---

# Database Principles

The Story Passport is the Aggregate Root.

Every major entity references:

story_passport_id

Avoid circular relationships.

Prefer explicit foreign keys.

Use Row Level Security consistently.

---

# Security

Never expose secrets.

Never hardcode credentials.

Validate every request.

Use least privilege.

Protect personal information.

Audit sensitive operations.

---

# Artificial Intelligence

AI assists engineering.

AI never replaces engineering judgment.

Generated code must always be reviewed.

Prompt history is not architecture.

The Company Brain defines architecture.

---

# Documentation

Every feature should update:

architecture documentation

API documentation

database documentation

decision logs (if needed)

README only when necessary

Documentation evolves with implementation.

---

# Testing Philosophy

Every feature should include:

unit tests

integration tests

manual validation

error handling tests

permission tests

AI workflow validation (when applicable)

Testing is mandatory.

---

# Code Review Checklist

Before merging:

Architecture respected

No duplicated logic

Types are correct

Security verified

Naming consistent

Tests passing

Documentation updated

Performance acceptable

No unnecessary complexity

If any answer is "No", revise before merging.

A feature is complete only if:

Business rules implemented

Architecture respected

Standards followed

Tests passing

Documentation updated

Security reviewed

Code reviewed

Ready for production

Anything less is incomplete.
```
