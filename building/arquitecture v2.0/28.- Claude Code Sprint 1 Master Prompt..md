# CTO Recommendation: A Better Execution Strategy

Ivan, after everything we've designed, I would make one significant change to how we work with Claude Code.

Instead of maintaining one massive prompt, I recommend creating an **Engineering Playbook** with three levels:

1. **Permanent Context** (never changes)

   * Company Brain
   * Product OS
   * Architecture Principles
   * Naming conventions
   * Coding standards

2. **Sprint Context** (changes every 1–2 weeks)

   * Sprint goals
   * Features to build
   * Features explicitly out of scope
   * Acceptance criteria

3. **Task Context** (changes every coding session)

   * One feature
   * One bug
   * One refactor
   * One UI screen

Claude Code performs best when it receives focused, bounded implementation work instead of an ever-growing master prompt. The Company Brain remains stable, the Sprint defines direction, and the Task defines the immediate objective. That structure will make development faster, easier to review, and much less prone to architectural drift as byDebut grows toward the Gemini XPRIZE MVP and beyond.
