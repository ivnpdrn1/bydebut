## ¿Qué colocas en Vercel?

Todo el proyecto generado por Claude Code:

```text
/app
/components
/lib
/hooks
/public
/styles
/package.json
/next.config.ts
/tsconfig.json
...
```

Es decir, **todo el proyecto Next.js**.

Cuando haces un cambio, Vercel:

* compila la aplicación,
* la publica,
* la deja disponible en Internet,
* administra el dominio,
* genera certificados HTTPS,
* realiza despliegues automáticos desde GitHub si lo configuras.

## ¿Cómo hablan entre sí?

Por APIs.

Por ejemplo:

```text
Usuario

↓

Vercel

↓

Next.js

↓

supabase.auth.signIn()
```

o

```typescript
const { data } = await supabase
  .from("books")
  .select("*");
```

En realidad eso envía una petición HTTPS a Supabase.

---

Otro ejemplo:

```typescript
const { data } = await supabase
.from("chapters")
.insert(...)
```

No escribe directamente en PostgreSQL.

Hace esto:

```text
Next.js

↓

HTTPS

↓

Supabase API

↓

PostgreSQL
```

# Hay un detalle importante

Dijiste:

> "Supabase actúa como BackEnd."

Sí, **en gran medida es correcto**, pero con un matiz importante:

* **Supabase** es tu **Backend-as-a-Service (BaaS)**. Proporciona la base de datos, autenticación, almacenamiento, funciones y APIs listas para usar.
* **Next.js**, que está desplegado en Vercel, también puede ejecutar **lógica de servidor** (API Routes, Route Handlers y Server Components).

Por eso, en una aplicación moderna, el "backend" suele repartirse así:

* **Supabase**: datos, autenticación, almacenamiento y servicios de backend.
* **Next.js en Vercel**: lógica de negocio, orquestación, validaciones e integración con APIs externas como Gemini o Stripe.
